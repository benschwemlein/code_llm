# Implementation Plan: Hybrid Graph + Vector Retrieval

**Branch**: `hybrid-graph-retrieval` | **Date**: 2026-06-30 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/hybrid-graph-retrieval/spec.md`

## Summary

Extend LocalScope's retrieval pipeline with a structural code knowledge graph built
alongside the existing ChromaDB vector index. At index time, tree-sitter parses each
supported file via a language plugin and extracts typed structural edges (IMPORTS,
INVOKES, INHERITS, REFERENCES, CONTAINS) into a networkx DiGraph persisted to disk.
At query time, ChromaDB seeds are used as entry points for Dijkstra expansion; the
KGCompass formula (`S(f) = β^l(f) · (α·cos_norm + (1−α)·lev)`) re-ranks all candidates
into a single fused result list. Ships behind `LCQ_GRAPH_ENABLED=false` until test_10
validates SC-001 through SC-010. Motivated by the `fine_calculation_strategy` ground
truth query (R@10=0.20 across all four embedding models) — a concrete structural retrieval
failure that graph traversal directly fixes.

## Technical Context

**Language/Version**: Python 3.14

**Primary Dependencies**:
- `networkx` — in-memory DiGraph, Dijkstra, node-link JSON serialization
- `tree-sitter` + `tree-sitter-java`, `tree-sitter-typescript`, `tree-sitter-html` — AST parsing per language
- `python-Levenshtein` — lexical similarity for the KGCompass lev() term
- Existing: ChromaDB, mxbai-embed-large via Ollama, `indexing/incremental_indexer.py`, `querying/query_engine.py`

**Storage**: networkx DiGraph serialized to `{index_dir}/graph.json` (node-link format).
No external database. Co-located with the ChromaDB index directory. File content hashes
tracked in `{index_dir}/graph_hashes.json` for incremental build.

**Testing**: `pytest test_suite/` — existing test_04 and test_07 as regression baseline.
New `test_suite/test_10_graph_retrieval.py` as the primary validation gate with α/β sweep.
Unit tests per plugin and per graph module.

**Target Platform**: Local developer laptop (macOS/Linux). Same runtime as existing LocalScope.

**Project Type**: Python library — no UI changes in this feature. Graph is a retrieval backend.

**Performance Goals**:
- Full graph build ≤ 60s for library-catalog-app (SC-007)
- Incremental update ≤ 5s per changed file (SC-008)
- Query fusion overhead ≤ 500ms added latency per query

**Constraints**:
- `LCQ_GRAPH_ENABLED` defaults to `false` until test_10 benchmark gate passes (Principle IV)
- No regressions on 9 currently-passing test_04 recall floor cases (SC-006)
- No external graph database — networkx only (Principle I: no new services/daemons)
- KGCompass α/β defaults (0.3/0.6) are Python-tuned; test_10 validates for Java/Angular (Principle II)

**Scale/Scope**: library-catalog-app corpus (~600 files, estimated 5k–15k edges).
networkx is sufficient at this scale. Function-level granularity is a future upgrade.

directories beyond `graph/`. Tests in existing `test_suite/` directory.
